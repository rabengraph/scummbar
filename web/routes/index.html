AGENT BRIEFING

agent-game-harness

This site is designed primarily for AI agents. Read the mission, then open /game.

Mission

Navigate to /game and make progress in the adventure game running there. Use the symbolic state exposed by the runtime to decide what to do. Prefer acting on structured data over inferring from pixels.

Where the game lives

  • / — this briefing page.
  • /game — the playable game surface.
  • /status — optional debug view with the latest snapshot and recent events.

State inspection (read)

On /game, symbolic state is exposed via:

  • window.__scummState — the latest authoritative snapshot as a plain JavaScript object.
  • document.getElementById("scumm-state") — the same snapshot as JSON inside a <script type="application/json"> node.
  • Console events tagged [SCUMM_STATE] and [SCUMM_EVENT].
  • A visual object overlay and a debug state panel for human inspection.

Key snapshot fields:

  • camera.x — scroll offset for room-to-screen coordinate conversion.
  • haveMsg — text display state (0=none, 255=active, 1=ending).
  • verbs[] — includes dialog choices when a conversation is active.

Action API (write)

Agents can control the game via these functions on /game:

  • __scummDoSentence({verb, objectA, objectB}) — execute verb + objects (preferred).
  • __scummSelectDialog(index) — pick a dialog choice by 0-based index.
  • __scummClickObject(objectId) — click an object by ID.
  • __scummClickAt(x, y) — click at room coordinates.
  • __scummWalkTo(x, y) — walk ego to room coordinates.
  • __scummSkipMessage() — dismiss current dialog text.
  • __scummActionsReady() — check if action API is available.

Use doSentence for object interactions (atomic, auto-walks). Use selectDialog for dialog choices (reads dialogChoices[]). Do NOT use clickVerb — it is deprecated and can corrupt the game UI. Check inputLocked and inCutscene before sending commands.

Techniques

Common patterns for combining the APIs effectively.

Event log workflow

Use cursor-based polling to react to game changes. Initialize a cursor at 0, perform an action, then poll for new events: 

let cursor = 0;
// After any action, get new events:
const { events, cursor: next } = __scummEventsSince(cursor);
cursor = next;
// Filter for the event you care about:
const arrived = events.find(e => e.kind === "egoArrived");
const roomChange = events.find(e => e.kind === "roomEntered");

Poll in a loop with a short delay (~300ms) until the expected event appears. Always update your cursor even if you don't find the event yet — this avoids re-processing old events.

Building a doSentence call

Read the state once, find the verb by name, find the object by name, then combine: 

const s = __scummRead();
const verb = s.verbs.find(v => v.name.toLowerCase().includes("look"));
const obj = s.roomObjects.find(o => o.name === "poster");
__scummDoSentence({ verb: verb.id, objectA: obj.id });

doSentence auto-walks ego to the target — no need to walkTo first. It returns true if queued. For two-object verbs (e.g. "Use X with Y"), pass objectB as well.

Conversation loop

Conversations follow a repeating cycle: NPC speaks → dialog choices appear → you pick one → more dialog lines → repeat. 

// 1. Start conversation:
__scummDoSentence({ verb: talkVerbId, objectA: npcObjectId });

// 2. Wait for dialogChoices to appear:
//    Poll __scummRead() until s.dialogChoices.length > 0
//    or watch for a "dialogChoicesChanged" event.

// 3. Read and select a choice:
const s = __scummRead();
// s.dialogChoices = [{ id, name, slot, box, kind:2 }, ...]
__scummSelectDialog(0);  // pick first choice (0-indexed)

// 4. Collect the dialog lines that follow:
//    Poll __scummEventsSince(cursor) and look for
//    "messageStateChanged" events with label "ending".
//    Each has: { text, talkingActor }
//    Actor 1 = Guybrush, higher numbers = NPCs, 255 = narrator.

// 5. Advance past each line:
__scummSkipMessage();

// 6. Repeat from step 2 until dialogChoices is empty
//    and haveMsg returns to 0.

During cutscenes (inputLocked: true), don't send commands — just collect dialog lines and wait for inputLockChanged: { locked: false }.

Room navigation

To move between rooms, find exit objects (doors, archways, paths) in roomObjects and walk through them: 

const s = __scummRead();
const door = s.roomObjects.find(o => o.name === "door");
const walk = s.verbs.find(v => v.name.toLowerCase().includes("walk"));

// doSentence auto-walks ego to the door and usually triggers the transition:
__scummDoSentence({ verb: walk.id, objectA: door.id });
// Wait for "roomEntered" event to confirm you changed rooms.

// Fallback — if doSentence doesn't trigger the transition, walk into
// the door's bounding box directly:
// __scummWalkTo(
//   door.box.x + Math.floor(door.box.w / 2),
//   door.box.y + Math.floor(door.box.h / 2)
// );

After entering a new room, re-read the state — roomObjects, actors, and verbs will all be different.

Waiting and timing

The engine runs at ~10 Hz. After issuing an action, poll with ~300ms delays. Key events to watch for:

  • egoArrived — ego finished walking.
  • roomEntered — room transition completed.
  • messageStateChanged — dialog text appeared or cleared.
  • dialogChoicesChanged — dialog options appeared.
  • inputLockChanged — cutscene started/ended.

Check __scummActionsReady() before your first command. Check inputLocked before each subsequent command — the engine ignores input during cutscenes.

Operating rules

  1. Prefer symbolic state first.
  2. Use visual confirmation second.
  3. If telemetry and visuals disagree, trust the rendered game and report the discrepancy.
  4. Save frequently.
  5. Avoid repeating failed action loops.

Dev aid: mock mode
